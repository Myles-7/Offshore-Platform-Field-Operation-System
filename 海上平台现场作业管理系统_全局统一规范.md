# 海上平台现场作业管理系统：全局统一规范

> 文件用途：本文件作为分模块编程前的全局统一约束文档，建议放在项目根目录或 `docs/GLOBAL_UNIFIED_SPEC.md`。  
> 适用对象：Codex、数据库开发、后端接口开发、PC 后台开发、Android 移动端开发、离线同步专项开发、AI 辅助验收开发、联调测试。  
> 使用原则：任何模块开发前必须先阅读本文档。若专项提示词与本文档冲突，以本文档为全局统一口径；若现有代码已有成熟规范，则在不破坏现有结构的前提下向本文档对齐。

---

## 0. 全局统一目标

本项目不是普通后台 CRUD 系统，而是面向海上平台、工业维修、防腐施工、设备检修等弱网现场的 **现场作业闭环管理系统**。

系统必须围绕以下主线实现：

```text
PC后台创建/分派工单
    ↓
移动端接收工单
    ↓
现场施工记录
    ↓
照片/视频/语音多媒体留痕
    ↓
照片水印与附件归档
    ↓
物料使用记录
    ↓
人员资质校验
    ↓
AI辅助识别疑似缺陷
    ↓
电子签名验收
    ↓
PDF验收单归档
    ↓
移动端离线数据同步到服务端
    ↓
PC后台复核、冲突处理、经营看板统计
```

所有模块都必须服务这条主线，不得把各模块做成互不关联的孤立功能。

## 1. 全局约束优先级

当 Codex 或开发者遇到多个文件、多个提示词或现有代码之间存在差异时，按以下优先级处理：

```text

```

说明：

1. 不允许为了套用本文档而破坏已有可运行项目结构。
2. 不允许因为某个单模块提示词而破坏全局主线。

4. 每个模块完成后必须说明与工单、同步、权限、文件、日志的关联情况。

# 第一部分：统一“核心业务主线”

## 2. 核心业务主线统一口径

### 2.1 一句话主线

本系统以 **工单 work_order** 为核心业务主线，将项目、人员、物料、施工记录、附件、电子签名、PDF 验收单、AI 识别结果、同步冲突和经营统计全部串联起来，实现海上平台现场作业从派工到验收归档的数字化闭环。

### 2.2 工单是系统中心

所有模块开发时，必须优先判断是否需要关联 `work_order_id`。

| 模块 | 是否必须关联工单 | 关联方式 |
|---|---|---|
| 项目信息 | 是 | `work_order.project_id` |
| 派工记录 | 是 | `work_order_assignment.work_order_id` |
| 状态流转 | 是 | `work_order_status_log.work_order_id` |
| 施工记录 | 是 | `work_order_record.work_order_id` |
| 多媒体附件 | 是 | `work_order_attachment.work_order_id` |
| 照片/视频/语音 | 是 | 附件表关联 `work_order_id`，可选关联 `record_id` |
| 电子签名 | 是 | `work_order_signature.work_order_id` |
| 验收记录 | 是 | `work_order_acceptance.work_order_id` |
| PDF 验收单 | 是 | `work_order_pdf.work_order_id` |
| 物料使用 | 是 | `work_order_material_usage.work_order_id` |
| AI 识别结果 | 是 | `ai_result.work_order_id`，同时关联照片附件 |
| 同步冲突 | 是/可选 | `sync_conflict.business_id` 或 `work_order_id` |
| 经营统计 | 是/可选 | `work_order_id`、`project_id`、`worker_id` 聚合 |

## 3. 统一工单生命周期

### 3.1 工单状态枚举

后端、数据库、PC 后台、移动端必须使用统一状态码。前端可以显示中文，但传输和存储建议使用英文枚举。

| 状态码 | 中文名称 | 说明 | 可操作角色 |
|---|---|---|---|
| `DRAFT` | 草稿 | 可选，工单未正式发布 | 调度员、项目经理 |
| `PENDING` | 待派工 | 工单已创建，未分派 | 调度员、项目经理 |
| `ASSIGNED` | 已派工 | 已分派给维修工或施工队伍 | 项目经理、调度员 |
| `ACCEPTED` | 已接收 | 维修工移动端确认接收 | 维修工 |
| `IN_PROGRESS` | 施工中 | 现场作业进行中 | 维修工 |
| `PENDING_ACCEPTANCE` | 待验收 | 施工完成，等待验收 | 维修工、验收人员 |
| `COMPLETED` | 已完成 | 验收通过并归档 | 验收人员、项目经理 |
| `REJECTED` | 已驳回 | 验收未通过，需要返工或补充 | 验收人员 |
| `CLOSED` | 已关闭 | 工单结束，不再流转 | 项目经理、系统管理员 |

### 3.2 状态流转规则

```text
DRAFT → PENDING → ASSIGNED → ACCEPTED → IN_PROGRESS → PENDING_ACCEPTANCE → COMPLETED → CLOSED
                                      ↓
                                   REJECTED → IN_PROGRESS / PENDING_ACCEPTANCE
```

规则：

1. 每次状态变化必须写入 `work_order_status_log`。
2. 状态变化必须记录操作人、操作时间、来源端、备注。
3. 移动端离线状态变化先写本地，联网后同步服务端。
4. 服务端处理离线状态同步时必须校验版本和当前状态是否允许流转。
5. 已完成、已关闭工单不得随意修改验收、签名、PDF 等关键归档数据。
6. 驳回必须填写原因。
7. 状态日志不得物理删除。

### 4.2 数据权限规则

1. 维修工只能访问自己的工单、施工记录、附件、签名、PDF 和 AI 结果。
2. 项目经理只能访问所属项目数据。
3. 物资管理员只能维护物料、库存、出入库、二维码相关数据。
4. 资质管理员只能维护员工档案、证书、证书预警。
5. 经营人员只能访问经营看板、统计报表和导出接口。
6. 系统管理员拥有全部权限。
7. 文件下载、PDF 下载、证书附件下载必须同时校验登录态和数据范围。
8. 权限不能只在前端控制，后端必须做强校验。

# 第二部分：统一“离线同步与冲突数据”

## 5. 离线同步总原则

移动端必须采用 **本地优先 Local First** 模式。

### 5.1 正确模式

```text
用户操作
  ↓
写入移动端本地 SQLite / Realm / Room
  ↓
写入 local_sync_queue
  ↓
页面立即反馈成功，并显示待同步状态
  ↓
网络可用时由 SyncWorker / WorkManager 批量同步
  ↓
服务端写入 MySQL
  ↓
服务端返回 server_id、version、sync_status
  ↓
移动端更新本地同步状态
```

### 5.2 禁止模式

```text
用户操作
  ↓
必须实时请求服务器
  ↓
服务器成功后才允许继续现场作业
```

该模式不适合海上平台弱网场景，禁止作为核心现场作业逻辑。

## 9. 统一同步接口

### 9.1 设备注册

```http
POST /api/sync/device/register
```

用途：移动端登录后注册设备。

请求核心字段：

```json
{
  "deviceId": "device-001",
  "deviceName": "Android-Phone",
  "platform": "ANDROID",
  "appVersion": "1.0.0"
}
```

规则：

1. `deviceId` 首次生成后长期稳定保存。
2. 退出登录不删除 `deviceId`。
3. 同一用户同一设备重复注册必须幂等。
4. 设备被禁用后拒绝同步。

### 9.2 心跳

```http
POST /api/sync/device/heartbeat
```

用途：更新设备在线状态和最后心跳时间。

### 9.3 增量拉取

```http
POST /api/sync/pull
```

用途：移动端拉取服务端变更数据。

必须支持：

1. 根据当前登录用户权限拉取数据。
2. 根据 `lastSyncTime` 或 `cursor` 增量返回。
3. 返回工单、所需物料、资质状态、检查项、知识库、冲突处理结果等。
4. 不返回图片、视频、语音、PDF 二进制，只返回文件元数据。
5. 首次同步也必须受权限范围控制。

### 9.4 增量上传

```http
POST /api/sync/push
```

用途：移动端批量上传本地变更数据。

必须支持：

1. 批量处理。
2. 部分成功。
3. 每条数据独立返回结果。
4. 根据 `local_id`、`server_id`、`version`、`updated_at` 判断新增、更新、删除或冲突。
5. 附件大文件不直接放入 `payload`，只上传附件元数据和 `file_id`。
6. 同一 `batchId` 重复上传必须幂等。

### 9.5 同步确认

```http
POST /api/sync/ack
```

用途：移动端确认某批 pull 数据已经成功写入本地数据库。

规则：

1. 移动端本地写入成功后才可 ack。
2. ack 失败不得删除本地数据。
3. 服务端记录 `sync_ack_record` 或等价日志。
4. ack 必须幂等。

## 11. 冲突处理统一规范

### 11.1 冲突判断规则

服务端处理 `sync/push` 时，按以下规则判断：

```text
1. server_id 不存在，local_id 未绑定服务端记录
   → 新增。

2. server_id 存在，客户端 version = 服务端 version
   → 正常更新，服务端 version + 1。

3. server_id 存在，客户端 version < 服务端 version
   → 可能冲突，进入字段比较。

4. 客户端删除，但服务端已更新
   → DELETE_AFTER_UPDATE 冲突。

5. 客户端更新，但服务端已删除
   → UPDATE_AFTER_DELETE 冲突。

6. 验收、签名、PDF 已锁定后再次被普通同步覆盖
   → ACCEPTANCE_LOCKED_CONFLICT。

7. 当前用户无权修改该数据
   → PERMISSION_CONFLICT。
```

### 11.2 冲突类型枚举

| 冲突类型 | 说明 |
|---|---|
| `VERSION_CONFLICT` | 版本冲突 |
| `FIELD_CONFLICT` | 字段级冲突 |
| `UPDATE_AFTER_DELETE` | 删除后更新冲突 |
| `DELETE_AFTER_UPDATE` | 更新后删除冲突 |
| `DUPLICATE_CREATE` | 重复创建冲突 |
| `PERMISSION_CONFLICT` | 权限冲突 |
| `FILE_META_CONFLICT` | 文件元数据冲突 |
| `ACCEPTANCE_LOCKED_CONFLICT` | 验收锁定冲突 |

### 11.3 冲突保存要求

冲突不能直接丢弃，必须写入 `sync_conflict`。

`sync_conflict` 至少包含：

```text
conflict_id
entity_type
entity_id
local_id
server_id
work_order_id，可选但建议
business_id
business_no
base_version
client_version
server_version
conflict_type
conflict_fields
client_payload
server_payload
resolved_payload
resolve_strategy
resolve_status
resolver_id
resolve_time
resolve_comment
device_id
operator_id
created_at
updated_at
```

### 11.4 冲突展示与处理

PC 后台必须提供冲突复核入口：

```http
GET  /api/admin/sync/conflicts
GET  /api/admin/sync/conflicts/{id}
POST /api/admin/sync/conflicts/{id}/resolve
```

处理策略：

| 策略 | 说明 |
|---|---|
| `KEEP_SERVER` | 保留服务端版本 |
| `KEEP_CLIENT` | 保留移动端版本 |
| `MANUAL_MERGE` | 人工合并 |
| `IGNORE_CLIENT` | 忽略客户端变更 |

处理规则：

1. 处理冲突必须填写处理说明。
2. 处理后更新业务表和版本号。
3. 处理后写入 `operation_log`。
4. 处理后移动端下次 `sync/pull` 能获取最终结果。
5. 已处理冲突不得重复处理，除非系统管理员重新打开。

## 16. 文件访问安全鉴权

### 16.1 禁止裸路径访问

```http
GET /api/files/{fileId}/preview
GET /api/files/{fileId}/download
```

### 16.2 鉴权规则

文件访问必须满足：

1. 用户已登录。
2. 文件未被逻辑删除。
3. 当前用户有权限访问文件所属业务对象。
4. 如果文件关联工单，则校验工单权限。
5. 如果文件关联项目，则校验项目权限。
6. 如果文件是证书附件，则校验资质管理权限或相关人员权限。
7. 如果文件是 PDF 验收单，则校验工单验收/项目/管理员权限。
8. 文件下载操作建议写入访问日志。

| 资质管理员 | 人员证书附件 |
| 经营人员 | 报表、对账单、统计导出文件 |

## 18. PDF 与签名安全规范

1. 签名数据必须绑定工单。
2. 签名应独立存储在 `work_order_signature`，不能只当普通附件。
3. PDF 验收单应独立存储在 `work_order_pdf`。

# 第六部分：统一 AI 辅助验收规范

## 28. AI 模块定位

AI 模块只能定位为 **辅助验收**，不能作为最终验收结论。

规则：

1. AI 结果只能提示疑似缺陷。
2. AI 结果不得直接改变工单最终状态。
3. 最终验收必须由人工确认。
4. AI 结果必须绑定工单、施工记录、照片附件。
5. AI 人工复核必须写入 `ai_review_record`。
6. AI 结果可进入 PDF 验收单摘要，但必须标注“辅助识别结果”。


## 33. 最终统一结论

本项目所有分模块开发都必须统一到以下四句话：

```text
1. 工单是主线，所有业务围绕工单闭环。
2. 移动端本地优先，所有弱网数据通过同步队列增量同步。
3. 文件只存元数据，访问必须鉴权，禁止裸路径和大文件入库。
4. API统一路径、统一返回、统一权限、统一状态枚举、统一错误码。
```

只要某个模块违反以上四条，就不能视为完成。
